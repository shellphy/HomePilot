# Neighborhood OS：产品哲学与领域模型
## Product Philosophy & Domain Model

> 2026-07-10。本文回答一个问题：如果这个产品不是装修工具、不是团购平台，而是一个真实小区的数字公共空间（Neighborhood OS），那么它最底层的模型是什么。
> 本文不讨论 UI、不讨论 MVP、不讨论商业模式。只讨论第一性原理。

---

# 第一部分：产品哲学（Why）

## 1. 从一个自然现象开始：为什么每个新小区都会长出微信群？

没有人规定要建群，但每个新交付的小区都会长出群，而且是在交房之前。为什么？

因为交房那一刻，几百个互不相识的家庭突然**共享同一个命运**：同一个开发商、同一个物业、同一堵墙、同一部电梯、同一个学区、同一张收费单。共同命运制造了两个需求：

- **共同的信息需求**——你家验房验出的问题，就是我家下个月的预告；你打听到的学区政策，就是我需要的答案。在小区里，一户的经验天然是全体的资产。
- **共同的行动需求**——面对开发商和物业，单个业主毫无议价能力；六百户一起，就是对方必须认真对待的力量。

微信群是这两个需求的**第一个自发制度**。人们每天在群里聊天，不是因为爱聊天，而是因为群是目前唯一的公共空间。聊装修、聊收房、聊物业、聊学区、聊停车——话题千变万化，底层关心的东西只有三件：**我最大的资产（房子）的价值、我日常生活的成本与摩擦、我在这个陌生地方的归属感。**

## 2. 群的天才与缺陷

群的天才：零门槛、即时、有人气。它完美解决**沟通**。

群的缺陷是结构性的，再多群管理也治不了：

| 群做不到的 | 后果 |
|---|---|
| **不沉淀**——一切被聊天流冲走 | 同一个问题每个月被重新问一遍；验房经验、成交价格、政策解读全部蒸发 |
| **不结构**——信息是文字流不是数据 | "多少户想装地暖"永远靠人肉数接龙 |
| **无身份**——群昵称≠这个小区的住户 | 名单不可信，投票无效力，商家混在业主里 |
| **无聚合**——个体表态无法变成集体事实 | 一百句"物业太差了"不构成一次有效力的集体意见 |
| **无边界**——群成员只是全体业主的子集 | 群里的"共识"只是 100 人的共识，另外 500 户不在场 |

一句话：**群是河，不是湖。** 河负责流动，湖负责蓄积。一个共同体不能只有河。

## 3. 重新定义对象：小区是一个未被数字化的共同体

600 户，约 2000 人——这是一个小镇的人口。这个"小镇"有：

- **经济**：亿级的装修季、每年数百万的物业费、公共收益（广告位、停车费）、内部交易（二手、拼车、互助）；
- **政治**：业主与开发商/物业的持续博弈、业委会的组建与选举、公共规则的形成；
- **社会**：从陌生到熟识的邻里关系网络、互助与信任的积累。

一个小镇会有档案馆、公告栏、议事厅、市场和名册。而这个数字时代的"小镇"，全部信息基础设施只有一个**会遗忘一切的聊天窗口**。

**Neighborhood OS 要做的，就是给这个共同体补上它缺失的公共基础设施——不是替代群，而是成为群背后的记忆、账本与议事厅。**

## 4. 使命的一句话表述

> **把散落的个体表态，变成可沉淀、可行动、可追溯的集体事实。**

拆开是三个词：
- **记忆（Memory）**——个体经验沉淀为社区档案；
- **协同（Coordination）**——分散意愿聚合为集体行动；
- **信任（Trust）**——透明的记录成为陌生邻居之间、业主与商家之间、业主与物业之间的信任基础。

装修团购只是这句话的第一个实例：登记（个体表态）→ 意向统计（聚合）→ 团购接龙（集体行动）→ 成交与评价（沉淀为集体事实）。同一句话可以实例化为收房维权、物业监督、业委会选举、邻里活动——这就是"OS"的含义。

## 5. 五条公理（设计原则）

**公理一：户是原子。**
人是流动的（业主会卖房，租客会搬走），房屋是恒定的。社区的权利单位是户（业委会投票按户和面积，不按人头），社区的历史挂在户上（这套房什么时候装修的、换过几任住户）。**数据模型必须以户为原子，人通过"与户的关系"获得身份。**"3栋老王"之所以可信，因为"3栋"可验证，"老王"只是称呼。

**公理二：与遗忘竞争，不与聊天竞争。**
OS 永远不做泛聊天，不追求日活，不制造内容流。群负责热闹，OS 负责记住。判断任何功能是否该做的测试：**它沉淀的东西，五年后还有人查吗？** 查——做；不查——留给群。

**公理三：结构化表态是唯一的价值来源。**
OS 里一切价值都来自居民做出的结构化表态（我登记、我参加、我投票、我评价）。表态必须绑定身份（户）、必须结构化（可聚合）、必须留痕（可追溯）。一万条聊天记录的价值小于一百条结构化表态。

**公理四：透明是内核约束，不是功能。**
公共空间里的一切集体运算——名单、票数、条件、账目、结果——默认可公示、可审计。这是 OS 与商业平台的根本区别：商业平台的算法是黑箱，公共空间的规则必须摊在阳光下。（透明不等于裸奔：见领域模型中的可见性分级——对外匿名聚合，对内身份可验。）

**公理五：OS 必须比它的创造者活得久。**
一个十年尺度的公共品不能以"某个热心业主"为单点。管理员是**角色**不是人；规则写在页面上而不是站长脑子里；数据属于社区、可导出、可移交（最终归宿可能是业委会）。从第一天起就按"有一天要交出去"来设计。

# 第二部分：领域模型（What）

## 6. 六个核心对象

整个系统只需要六个一等对象。所有场景都是它们的组合。

### ① 户（Unit）
社区的空间原子与权利单位。约 600 个，从交付起恒定存在。
关键属性：楼栋/单元/户型；权利属性（投票权重、面积）。
**户是一切记录的最终挂靠点**：这套房的装修史、参与史、住户变更史。

### ② 成员（Member）＝ 人 × 户 × 关系
人不直接存在于社区中，人**通过与户的关系**存在：业主、家属、租客。同一个人可以关联多户（买了两套），一户可以有多个成员。
关键属性：关系类型、**验证等级**（未验证 / 邻居担保 / 凭证验证）。验证等级决定权利：未验证可看公开信息，验证业主才能投票。
**身份即关系，权利即验证等级**——这是整个信任体系的地基，也是群永远给不了的东西。

### ③ 相关方（Party）
与社区发生关系的组织：开发商、物业、商家、业委会、居委会。
它们不是"用户"，是**对手方与服务方**：拥有累积的信用档案（评价、履约记录），拥有受限的可见范围（商家能看聚合需求，看不到名册）。
Party 的信用档案是社区最有权力的资产之一——"这家干砸过一单，600 户都查得到"。

### ④ 事务（Matter）
**系统的中心运行时对象**：社区正在处理的一件事。装修团购是事务，电梯维修是事务，业委会筹备是事务，中秋活动也是事务。
事务有**类型**（决定它装载哪些能力）和**生命周期**（提出 → 征集 → 行动 → 结果 → 归档）：

| 事务类型 | 例子 | 装载的表态模式 | 权利规则示例 |
|---|---|---|---|
| 集体采购 | 装修团购、团菜 | 接龙 + 评价 | 验证户可参加 |
| 互助 | 拼车、借物、宠物代喂 | 接龙 | 成员即可 |
| 治理 | 物业费议题、公约、业委会 | 登记 + 投票 | 一户一票/面积加权，凭证验证 |
| 维权 | 收房质量联名、公共收益追讨 | 接龙（联名）+ 登记（问题征集） | 验证业主 |
| 活动 | 邻里节、球局 | 接龙 | 成员即可 |
| 公告 | 物业通知、政策转达 | 无（单向）+ 评论可选 | 发布权受限 |

现有装修产品里 project 的 seeking→negotiating→open→done，只是"集体采购"类事务生命周期的一个实例——**装修没有被推翻，装修被收编了**。

### ⑤ 记录（Record）
系统的沉淀原子：**（谁[成员/户]，何时，就何事[事务/相关方/户]，做了什么表态/留下什么事实）**。
四种表态模式其实是一个引擎的四档，恰好覆盖一件事的完整周期：
- **登记**（事前）——申报自家的事实与意愿："我家 107 户型，想装地暖"；
- **接龙**（事中）——作出可数的参与承诺："算我一户"；
- **投票**（事中）——在选项间行使决策权："同意换物业"；
- **评价**（事后）——对结果和相关方作出判断："这家门窗，四星，打胶毛糙"。
记录的三个铁性质：**绑定身份**（挂靠户）、**只增不改**（修正以新记录追加，历史可查——信任的技术基础）、**可见性分级**（本人可见 / 管理者可见 / 匿名聚合公示 / 实名公示，在创建时确定并告知）。

### ⑥ 空间（Place）
物理公共物的索引：楼栋、电梯、车库、会所、绿地。不是行为主体，是**记录和事务的挂靠坐标**（"3 栋 2 单元电梯"的维修史）。让"地图"成为浏览社区记忆的另一个维度。

**对象间的关系一句话**：成员（人×户）就事务和相关方作出记录，事务围绕空间与相关方展开，一切记录最终沉淀到户、相关方与空间的档案里。

## 7. 什么值得沉淀：资产判别

用公理二的五年测试过滤：

**资产（湖，OS 的领地）**：
1. **名册**——户-成员关系图谱。社区一切集体行动的地基，也是最敏感的资产（分级可见，永不整体导出给任何 Party）；
2. **事务档案（先例库）**——每件事从提出到结果的完整记录。"上次电梯问题怎么解决的"“2027 年门窗团购什么价"——新业主的教科书，老业主的判例法；
3. **信用账本**——对商家、物业、开发商的累积评价与履约记录。时间越久越有权力；
4. **交易事实**——真实成交价、费用、账目。信息不对称的解药；
5. **本地知识**——验房怎么验、学区怎么申请，**限"本小区的具体经验"**（通用知识让给小红书和 AI）；
6. **规约**——社区自己形成的规则、公约、议事惯例。

**气泡（河，留给微信群）**：寒暄、即时求助、情绪、通知的送达（群转发是最好的通知通道）。OS 不吸收气泡，只在气泡即将凝结成事实时提供容器（"这事别在群里说了，发起一个事务吧"）。

## 8. 能力内核：三层十件

**表达层**（个体如何输入）
1. 表态引擎（登记/接龙/投票/评价 四模式，一套实现）
2. 评论（刻意最弱化——事务内的轻讨论，深聊回群）

**组织层**（集体如何处理）
3. 事务生命周期状态机（类型即模板：每种事务类型预置表态模式+公示模板+权利规则）
4. 公示渲染（名单、进度、票数、账目、结果的标准呈现——公理四的实现）
5. 时间线（单一事务的经过 + 全社区的大事记）

**基础层**（系统性能力）
6. 身份与权利（户验证、关系管理、验证等级、每类事务的参与权规则——**一户一票还是一人一票必须是可配置参数**，因为不同事务的合法性来源不同）
7. 通知（订阅消息 + 可转发回群的卡片——转发即通知）
8. 搜索（对全部沉淀的检索）
9. **AI（社区记忆的自然语言界面）**——AI 在 OS 里的正确定位终于清晰：不是内容生产者，不是聊天机器人，而是**档案馆的问询台**——"咱们小区装地暖的多吗？大概什么价？"答案来自记录与档案，引用可溯源。没有沉淀的 AI 是套壳，有了沉淀的 AI 是唯一读得完全部档案的馆员；
10. 空间索引（记录挂坐标，按楼栋/设施浏览）

**组合公式**：
> **场景 ＝ 事务类型 ＋ 表态模式组合 ＋ 公示模板 ＋ 权利规则**

新场景 = 一行配置，不是一次开发。这就是"针对装修单独开发"和"OS"的区别。

## 9. 场景装载验证：所有话题跑一遍内核

用户列举的全部话题，无一需要新原语：

| 话题 | 映射 |
|---|---|
| 装修团购 | 集体采购事务 + 接龙/评价 + 名单进度公示（现有产品 ≈ 该类型的硬编码实现） |
| 装修意向 | 登记记录（挂户）+ 匿名聚合公示（现"小区意向"页） |
| 收房/验房 | 维权/互助事务 + 登记（问题征集，挂户与空间）+ 本地知识沉淀 |
| 开发商/物业 | Party 信用账本 + 治理/维权事务 + 评价 |
| 业委会 | 治理事务 + 投票（一户一票/面积加权 + 凭证验证）+ 公示 |
| 学区/政策 | 本地知识 + 公告事务 + 登记（"谁家孩子明年入学"聚合） |
| 停车/公共空间 | 空间对象 + 治理事务 |
| 邻里活动 | 活动事务 + 接龙 |
| 二手/拼车/拼团 | 互助/采购事务 + 接龙（轻权利规则） |
| 社区公告/事件 | 公告事务 + 时间线（大事记） |

反向检验（内核封闭性）：有没有内核表达不了的社区事务？纠纷调解（人与人矛盾）——刻意不做，OS 不当法官，最多沉淀"事实记录"；隐私性个体服务（家政预约）——不做，那是商业平台的事。边界清晰。

## 10. 十年问题与北极星

**这个产品五年、十年后还存在吗？** 把社区的生命周期摊开：

交付期（收房维权、装修）→ 安顿期（入住、配套、学区）→ 治理期（业委会、物业博弈、公共收益）→ 成熟期（设施老化维修、邻里网络、更新改造）。

每个时期的**主导事务类型**在变，但事务、表态、沉淀、信任这四件事永远在发生。装修季会结束，**共同体不会结束**。OS 跨越所有时期的条件只有一个：内核对（六对象十能力），而不是某个场景做得深。十年后它的形态：一个新业主搬进来，打开它能回答——这个小区经历过什么、哪些商家可信、物业换过几次、我家这套房的前世今生、现在正在发生什么、我可以加入什么。**它是这个小区的档案馆、议事厅与名册。**

**北极星**：

> **社区公共记忆的完整度——凡小区之事，皆有档可查；凡集体行动，皆有始有终。**

可度量的形态：**每年经由 OS 走完完整生命周期（提出→行动→结果→归档）的公共事务数 × 参与过表态的户数占比（/600）。** 前者度量"议事厅是否在运转"，后者度量"这是全体的公共品还是少数人的工具"。

它统一了此前所有版本的北极星：装修季的"真相覆盖率"只是它在第一个场景的投影。它天然抵制虚荣建设（内容、日活、功能数对它无贡献），天然指向正确的行为（把群里将要蒸发的事实接进容器，把散落的意愿组织成有结果的行动）。

**十年检验（考古测试）**：2036 年，一个刚买了二手房的新业主问它："这个小区值得住吗？物业怎么样？当年装修哪家口碑好？"——如果它答得出来，这十年它就没有白活。
