## 一、目标 Schema（全新迁移，删除全部旧域迁移）

- **units**：id, label(unique), timestamps —— 户（粒度渐进：现在=楼栋，将来可细化）
- **parties**：id, type(merchant/property/developer/committee), name, category(''), is_listed(false), timestamps —— 相关方
- **residents**：id, openid(unique), nickname(''), avatar(''), wechat_id(''), phone(''), unit_id(FK null), party_id(FK null), timestamps —— 成员；商家=绑定 party，废除 role/merchant_* 字段（API 层派生 role 保持前端兼容）
- **matters**：id, type, initiator_id(FK null), party_id(FK null), title, category(''), state, is_approved(false), target_count(0), payload(json), timestamps —— 事务；groupbuy 的 pitch/perk/terms/glossary/final_terms/final_note 入 payload
- **records**：id, matter_id(FK null), party_id(FK null), resident_id(FK), mode(register/join/review，将来 vote), subject(''), payload(json), timestamps —— 表态；装修档案= matter_id null + mode=register + subject=renovation（layout/decoration_mode/interests/answers 入 payload）；唯一性由 updateOrCreate/firstOrCreate 在应用层保证
- **record_revisions**：id, record_id(FK), payload(json), created_at —— 只增不改的修订链（修改表态时把旧 payload 存档）
- **matter_updates**：id, matter_id(FK), happened_on(date), content, images(json), timestamps —— 事务时间线（原 progress_updates）

删除模型/表：Project、Signup、Registration、Review、ProgressUpdate。ProjectStatus 枚举删除（状态机进类型类）。

## 二、类型引擎 `app/Matters/`

- `MatterType` 抽象类：key()/label()/states()[state=>label]/initialState()/payloadRules()/sortWeight(Matter)/allowsJoin()/allowsReview()
- `GroupbuyType`：states seeking意向征集/negotiating谈判中/open接龙中/done已成团；允许 join；done 且参团者可 review；列表排序 open>negotiating>seeking>done
- `NoticeType`：states published/archived；payload=body；不可 join/review —— 第二类型，证明"新场景=一个类"
- `MatterTypeRegistry::for($type)` / `keys()`

## 三、API（v2，小程序同步更新）

不变：POST /login、GET /options、GET/PUT /me、GET/PUT /registration、GET/PUT /survey、GET /stats、POST /uploads（registration/survey 内部写 records）
新增/替换（原 /projects 全套换名）：
- GET /matters（已审核，混类型，按类型 sortWeight 排序）/ GET /matters/mine / GET /matters/{id}（附 joined、my_review）
- POST /matters（type 必填，按类型校验 payload）/ PUT /matters/{id} / PUT /matters/{id}/state / PUT /matters/{id}/deal（groupbuy 专属）
- POST|DELETE /matters/{id}/join（原 signup）/ PUT /matters/{id}/review
- POST /matters/{id}/updates（原 progress）
- POST /me/party {name,category}（商家入驻=创建并绑定 merchant party）/ DELETE /me/party（切回业主）

MatterResource 输出：type/type_label/state/state_label/target_count/join_count + groupbuy payload 平铺（pitch/perk/terms/glossary/final_terms/final_note）或 notice 的 body；roster/updates/reviews 同前。ResidentResource 派生 role（有 merchant party 即 'merchant'）+ merchant_name/category，前端逻辑少改。

## 四、Filament 重建

MatterResource（审核/编辑 + updates/join 关系管理器）、RecordResource（档案查看，替代 Registrations）、PartyResource（公示商家管理）。

## 五 B、小程序结构重构（2026-07-10 追加，与后端对称）

目标：**类型即组件**——新事务类型接入 = 新增一张卡片组件 + 一个详情组件 + 列表/详情页各一行 elif，页面永远是薄壳。

```
miniprogram/
  utils/api/          # 服务层：页面不再拼 URL
    matters.js        # listMatters/getMatter/createGroupbuy/updateGroupbuy/flipState/publishDeal/join/leave/review/postUpdate
    profile.js        # getOptions/getStats/getRegistration/saveRegistration/getSurvey/saveSurveyAnswers
  behaviors/load.js   # 页面加载模式复用：首载骨架/错误重试/静默刷新（runLoad）
  components/
    page-status/      # 加载态壳（slot + 骨架 + 错误重试），全页面复用
    groupbuy-card/  notice-card/       # 列表卡片，按类型
    groupbuy-detail/  notice-detail/   # 详情体，按类型，自带该类型的全部行为（报名/评价/流转/成交公示入口）
  pages/              # 页面改名对齐领域模型，全部变薄壳
    community/  (原 groupbuys，tab 小区)   insights/ (原 map，tab 意向)   my/
    matter/     (原 project，详情壳+类型分发)
    groupbuy-form/ (原 leader-project)   matter-update/ (原 leader-progress)   groupbuy-deal/ (原 leader-deal)
    registration/  survey/  (名字保留)
```

要点：组件 styleIsolation 用 apply-shared（沿用 app.wxss 的全局样式系统）；卡片/详情组件内部自算 pillClass/percent/stars 并自持行为，向页面只发 refresh 事件；分享、导航标题留在页面壳上。
