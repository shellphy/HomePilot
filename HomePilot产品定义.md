HomePilot 产品定义（V0.3）

> V0.3 基于四个明确约束重写：
> ①精力有限，产品必须极简；②永远只服务本小区，不做推广；
> ③长期希望长成小区公共服务工具；④第一阶段目标是组织团购，团长是我本人。
>
> V0.1/V0.2 的"AI 装修画像 / 认知工具"方向存档不再作为核心（见文末存档说明）。

⸻

一、一句话定位

**「本小区的装修团购工具」**——一个把微信群里混乱的团购组织过程结构化的小程序。

微信群解决不了的问题，就是这个工具存在的理由：

* 群消息一刷就没，报名信息散落各处，统计靠手动
* 谁在什么装修阶段、对什么感兴趣，完全不透明
* 团购进度（报了多少户、谈到什么价）没有一个公示的地方
* 新入群的业主看不到历史信息，同样的问题反复问

⸻

二、运营策略：公开的团长模式

工具只是载体，团购的核心是信任。策略是**把团长身份完全摆在明面上**：

1. **自家做样板**：我自己家先和选定的装修公司签约，全程公开——工地进度照片、报价明细（脱敏）、遇到的问题都在小程序里晒出来。样板间效应是最强的转化。
2. **公开发起团购**：以团长身份发布"我做了功课、我自己家就用这家、我谈下的团购条件是 X，愿意跟的报名接龙"。
3. **进度公示制造势能**："已有 17 户报名"本身就是最好的推动力，工具负责把这个数字实时可见。
4. **团长条件半公开**：和商家谈的团长权益（自家折扣/服务费）属于行业常态，不需要广播，但不否认、不宣称纯公益。

明确不做的事：不做伪中立的"AI 推荐"，不假装客观分析然后指向特定商家。工具的公信力 = 团长的公信力，透明是唯一可持续的玩法。

⸻

三、MVP 功能（极简，三个页面）

**1. 团购项目列表（首页）**

每个项目一张卡片：品类（装修公司/中央空调/地板/全屋定制…）、当前状态（意向征集中/谈判中/已成团）、已报名户数。

**2. 项目详情页**

* 团长发布的内容：方案介绍、报价、进度更新（时间线形式）
* 报名接龙：一键报名，列表可见（楼栋+昵称，不显示敏感信息）
* 样板记录：装修公司项目里挂我自己家的施工记录
* **「买前必懂」微科普卡片**：每个团购品类 3~5 条，只讲影响这次决策的概念（如中央空调页讲匹数估算、变频压缩机、内机检修口；瓷砖页讲规格、釉面、吸水率）。固定格式"术语 → 大白话 → 对你的影响"。内容用 AI 生成初稿、我把关一次成型，不做持续运营。群里反复被问的问题也沉淀到这里，答一次管所有人

**3. 装修意向登记（轻量表单，2 分钟）**

四题：户型（本小区就 107/130/154 三种）、装修方式（全包/半包/清包/还没定——小区同期交付，大家同期装修，**不问开工时间**）、感兴趣的品类（多选）、联系方式（**楼栋号必填、微信号必填、手机号选填**——房号不收集，保护隐私；昵称头像用微信原生组件在"我的"页获取）。

汇总输出一张"**小区装修进度地图**"：各户型有多少户登记、开工时间分布、各品类意向人数——这既是给所有业主的公共信息（比群里问一百遍强），也是团长发起下一个团购的依据、和商家谈判的筹码。

**4. 发起制 + 审核制（2026-07-10 修订：取消"全局团长"）**

* **人人可发起**：任何业主都能发起团购，发起人即该团团长，负责对接商家、更新进度、流转状态。入口在团购列表底部和"我的"页。
* **管理员审核**：发起的团购默认"待审核"，不对外展示；管理员在 Filament 后台审核上架。发起人在"我的 → 我发起的团购"里能看到自己项目和审核状态。
* **发起表单**：品类（可自定义）、标题、初始状态（先征集意向 / 直接开接龙，对应"没谈价先探人数"和"条件已谈好"两种节奏）、目标户数、团购条件（可后补）、买前必懂。
* **状态机**：意向征集 → 谈判中 → 接龙中 → 已成团，由发起人手动流转；进度更新（文字+现场照片）落在详情页时间线上。
* **铁律：小程序不碰钱**。不代收款、不担保交易，报名仅为意向登记，签约付款由业主直接对商家——避开支付资质、资金池责任和售后纠纷，这条永久有效。

**5. 问卷体系一期（2026-07-10 修订：从"不做"清单中捡回）**

* **两层结构**：基础登记（5 题，2 分钟，登记即有用）+ 进阶问卷（4 模块 12 题：家庭与居住 / 生活方式 / 收纳与家务 / 预算与担忧，约 3 分钟，可跳题、可分次填）。
* **题目由服务端下发**（后端配置文件定义题库），迭代问卷不需要发版小程序；答案以 {题目key: 答案} 存 JSON，按模块提交、增量合并。
* **数据用途**：匿名统计进小区进度地图（后续接入）、管理员在后台看明细，作为团购谈判筹码和内容规划依据。问卷原则沿用 V0.1：生活方式优先、不用行业术语、像聊天不像考试。
* AI 画像仍不做；问卷价值先通过"统计给团长当弹药"兑现。

**6. 商家身份（2026-07-10 增补）**

群里的商家可以在"我的"页自助切换为商家身份（商家名称+主营品类，带"商家"标识）：

* **能看**：团购列表、详情、小区意向统计——"23 户对中央空调感兴趣"对商家就是鱼饵，这正是谈判势能的来源；
* **不能干**：报名接龙、填业主登记、发起团购——保证意向数据全部来自真业主；
* 商家报价/接单等交互留二期，现阶段商家想合作走线下找管理员。

**明确不做**：AI 画像、独立知识库/百科（科普只以「买前必懂」卡片挂在团购页上，不设单独板块）、社区、商家报价/交易（只有商家身份浏览）、评论区（评论引战风险大且需要运营精力，讨论留在微信群）。

⸻

四、通往"小区公共服务工具"的路

不为未来多写一行代码，但 MVP 的三个基础件天然可以泛化：

```
登记表单  →  拼车登记、闲置置换登记
接龙报名  →  任何集体行动（团购快递柜、联名投诉、活动组队）
进度公示  →  物业报修追踪、公共事务进展
```

装修期（未来一两年）用团购功能建立用户基础和习惯，装修潮退去后，同一套"项目+接龙+公示"的骨架换品类继续用。所以数据模型里"项目"的品类字段设计成开放枚举即可，其他一概不提前设计。

⸻

五、冷启动

顺序是**用第一个真实团购带工具，而不是先推工具**：

1. 我自己家先定装修公司，开始记录样板素材
2. 小程序只做三个页面，上线
3. 在群里发起第一个团购（装修公司），报名入口指向小程序
4. 报名的人顺手完成意向登记 → 进度地图开始有数据 → 第二个团购（中央空调等）的弹药就有了

⸻

六、验证标准

* 第一个团购通过小程序报名的户数（>15 户即成立）
* 意向登记完成数（>30 户说明"进度地图"有公共价值）
* 群里出现"去小程序看"的自发引用（说明工具开始替代群聊的某些功能）

⸻

七、技术选型（2026-07-09 定）

* **前端**：uniapp（Vue3 + vite）。日常开发用 H5 编译目标在浏览器里测试（Claude 可用 playwright 全流程验证），发布前在微信开发者工具和真机人工过一遍微信特有环节（登录、订阅消息）。业务逻辑抽成不依赖框架的纯 JS 模块，用 vitest 单测。
* **后端**：Laravel API（本人熟悉的生态），6~8 个接口：登录（code2session 换 openid）、项目列表/详情、报名、意向登记、统计汇总、团长发布/状态流转。测试用 Pest 全覆盖。
* **数据库**：MySQL；若服务器内存只有 1G 则用 SQLite（本项目规模下完全够用）。
* **部署**：阿里云轻量应用服务器（2 核，内存尽量 2G）+ 已备案域名 + HTTPS（免费证书）。
* **备案清单**：①域名 ICP 若非阿里云接入，需做"新增接入"；②小程序本身需在微信公众平台完成小程序备案（2023 年新规），注册账号后立即提交、与开发并行。
* 放弃微信云开发的原因：云函数/云数据库的集成测试依赖真实云环境和开发者工具 GUI，Claude 难以代为验证；Laravel 在本地可被完整地端到端测试，且是本人最熟的技术栈。

⸻

八、存档说明

V0.1/V0.2 的核心思路是"AI 装修画像 + 认知陪伴工具"，放弃原因：

* 它是整个设计里最耗精力的部分（问卷体系、Prompt 调试、户型预研），与"精力有限"冲突
* 它服务的目标（帮用户慢慢建立认知）不是当前第一优先级（组织团购）
* 其中有价值的残留已并入 V0.3：「邻居镜子」瘦身为"装修进度地图"；「站在业主角度、不用专业术语」原则继续适用于所有页面文案

如果未来团购稳定运转、又确实观察到大量业主卡在"不知道自己要什么"，可以再把画像作为独立功能捡回来。
